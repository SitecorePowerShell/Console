**Severity re-scoped after empirical measurement (2026-04-18):**
Original framing ("billion-laughs DoS") overstated the impact on this stack.
Measured behavior of the pre-fix code on .NET Framework 4.8:
- XmlTextReaderImpl has a hardcoded MaxCharactersFromEntities = 10,000,000,
  so catastrophic memory exhaustion cannot occur; a depth-7 fan-out-10
  payload (30M expected chars) throws "The input document has exceeded a
  limit set by MaxCharactersFromEntities" after ~1s.
- Depth 6 (3M chars) deserializes successfully in ~1.15s and allocates
  ~6 MB of string per request. Depth 5 (300K chars) completes in ~166ms.
  This is a real but sub-catastrophic CPU/allocation amplification: a
  ~700-byte payload forces ~1s CPU and several MB of RAM per request.
- Single-entity substitution works silently (`<!ENTITY xxe "PWNED">` +
  `<S>&xxe;</S>` deserializes to "PWNED"), confirming DTD processing is
  active on an endpoint that never legitimately receives a DOCTYPE.
- The right fix is still at the API layer (DtdProcessing = Prohibit)
  rather than relying on an internal framework cap that could move in a
  future .NET update.

## Planned (needs additional research): Invoke-RemoteScript API-key feature gaps (auto-retry + observability + diagnostics)

- [ ] Phase 1: Plan - needs research before committing to an approach
- [ ] Phase 2: Branch, TDD red tests first (all scenarios below must fail on current code before any implementation), green implementation
- [ ] Phase 3: Merge

**TDD rule for this entry:** Every gap below requires a failing test (Pester integration or unit) landing BEFORE the corresponding fix. No behavior change without a red test that proves the gap exists today. This applies to each individual gap - don't bundle the fixes ahead of their tests. Test sketches listed per-gap below.

**Planned branch:** feature/remoting-client-api-key-gaps (scheduling TBD)

**Observed behavior today:**
Invoke-RemoteScript.ps1:407-428 reads Retry-After, X-RateLimit-Limit, and X-RateLimit-Reset headers on HTTP 429, embeds the values in a Write-Error message ("Rate limit exceeded. Limit: N... Retry after N seconds."), sets $encounteredError, and propagates. No Start-Sleep, no retry loop, no backoff. The only Start-Sleep calls in the module are connection warm-up (line 199) and polling in Wait-* / Stop-ScriptSession - all unrelated. Beyond 429, the client also lacks handling for 503 cold-start retries, 401 expired-key diagnostics, success-path rate-limit visibility, and X-RateLimit-Remaining parsing.

**Gap 1 - Auto-retry on 429 (known)**
- Rough direction: opt-in retry via a new -MaxRetries / -RespectRetryAfter parameter set defaulting safely; wrap the send in a for-loop that honors Retry-After on 429, capped at a sane ceiling (e.g. 60s) to prevent a hostile server parking the client; also retry transient 5xx with exponential backoff.
- TDD red test sketch: use a real API Key with RequestLimit=1/window, issue two requests back-to-back with -MaxRetries=1. Today: second request fails with "Rate limit exceeded" error. After fix: second request sleeps Retry-After seconds then succeeds. Test asserts final status and total elapsed >= Retry-After.

**Gap 2 - Cold-start 503 auto-retry (medium)**
- Server emits 503 + Retry-After: 2 at RemoteScriptCall.ashx.cs:244-246 when the API key registry hasn't loaded yet. Client has no 503 branch; first-request-after-restart fails with a generic error instead of transparently waiting and retrying. This is the one case the server explicitly asks the client to auto-retry.
- Fix direction: add a 503 branch parallel to the 429 branch. Honor Retry-After (default 2s if missing). Shorter ceiling than the 429 case - cold start is expected to resolve in seconds, not minutes.
- TDD red test sketch: restart the CM container (or force an authorization-cache invalidation), then issue a request immediately. Today: 503 returns as a generic connection error. After fix: client sleeps and retries transparently, request succeeds on retry. If container restart is too heavy for integration tests, a mock HTTP server that returns 503 once then 200 works instead.

**Gap 3 - 401 expired-key diagnostics (low-medium)**
- Server rejects expired keys with 401 (integration test "[PASS] Expired key is rejected (401)" at line 1522 confirms this). Client's catch [HttpRequestException] block only handles 429 and 403. 401 falls through to the generic error path; admins debugging expired keys see "Server response: Unauthorized" rather than "API Key expired".
- Fix direction: add a 401 branch. Inspect response body (server may include a reason payload) and/or WWW-Authenticate header. If the server doesn't currently surface the expiration reason to the client, add a server-side header (e.g. X-SPE-AuthFailureReason: expired|disabled|signatureInvalid) and handle it on the client.
- TDD red test sketch: create an API Key with Expires=past-date, issue a request with its shared secret. Today: error message is generic "Unauthorized". After fix: error message says "API Key expired on <date>" or equivalent, with a distinguishable ErrorCategory (AuthenticationError vs ConnectionError).

**Gap 4 - Rate-limit headers on successful responses not surfaced (low, observability)**
- Server emits X-RateLimit-Limit / -Remaining / -Reset on every API-key-authenticated request (success or 429) via AddThrottleHeaders at line 141. Client only reads them inside the 429 error branch. Successful callers never see remaining budget, can't preemptively slow down.
- Fix direction: on successful response, read the rate-limit headers and expose them. Two options:
  - Attach to the returned object as properties (works for Invoke-RemoteScript return value shape).
  - Emit via Write-Verbose -Message when the headers are present (non-intrusive, opt-in via -Verbose).
- Decision needed in Phase 1: changing return shape is a breaking contract; Write-Verbose is safe but less discoverable.
- TDD red test sketch: issue a successful request with an API Key that has RequestLimit=60. Today: no rate-limit info reaches the caller. After fix: caller can read (via property or -Verbose capture) Limit=60, Remaining=59, Reset=<timestamp>. Integration test asserts values match what the server sends.

**Gap 5 - X-RateLimit-Remaining dropped on the floor (trivial)**
- Even in the 429 branch, Invoke-RemoteScript.ps1:414-418 reads only X-RateLimit-Limit and X-RateLimit-Reset. X-RateLimit-Remaining is never parsed, so the error message says "Limit: M" instead of "exceeded by N over cap M".
- Fix direction: one-line add of the header read + message concat. Trivially bundled with Gap 1 and Gap 4.
- TDD red test sketch: exceed the limit on a throttled API Key with a known Limit. Today: error message does not contain the word "Remaining" or the remaining count. After fix: error message surfaces remaining=0 (or the overshoot count).

**Open questions that require research before implementation (apply to Gaps 1 and 2):**
1. Default on or off? Auto-retry can mask abuse and hide client bugs; defaulting off preserves current behavior but fewer consumers benefit. Need to check how other SPE clients (MCP server) expect to handle 429/503 today.
2. Which status codes should trigger retry? 429 clearly. 503 clearly (server asks for it). 502/504 maybe. What about 500? (Generally no - retry on 500 can amplify real server bugs.)
3. Retry-After parsing: header can be either seconds (integer) or an HTTP-date. Today's code treats it as a string; a robust implementation needs to handle both forms.
4. Ceiling on Retry-After honoring - hostile server could send Retry-After: 86400. 60s for 429, maybe 10s for 503 as starting suggestions; confirm with operators.
5. Idempotency concern: retry is safe for read-only scripts but dangerous for scripts that mutate state. Caller-script semantics are opaque to the client. Either restrict retry to known-safe operations (file download, GET-style requests) or document that enabling retry means the caller owns idempotency.
6. Interaction with session-persistence (-PersistentSession). If a retry lands after the session has expired, behavior diverges. Needs test coverage.
7. Backoff algorithm: fixed Retry-After, exponential with jitter, token bucket? Exponential + jitter is standard; Retry-After should be honored as a lower bound, not overridden.
8. Cancellation: long retry sleeps need to honor Ctrl+C / cancellation tokens.
9. Test harness: need a way to simulate 429/503 responses in integration tests. Today's throttle tests exercise the server side; a client-retry test needs either a mock HTTP server or a real throttled API Key with very low limits to trigger 429 predictably, and (for 503) either a cold-start orchestration or a mock.

**Additional open question for Gap 3:**
10. Does the server currently surface expiration reason in a consumable form, or does this require a matching server change (new header X-SPE-AuthFailureReason)? If server change is needed, scope both sides in the same branch to avoid a partial-rollout window.

**Additional open question for Gap 4:**
11. Return-shape change vs Write-Verbose: which do consumers prefer? Check how the MCP server and any existing Invoke-RemoteScript callers consume the return value today. A property-bag attachment could break strict-typed consumers.

**Why not yet scheduled:** Design questions above have non-obvious answers that depend on consumer expectations (MCP server, CI/CD pipelines, interactive ISE). Worth a short research spike before committing to a plan. Gaps 1, 2, 3 need design; Gaps 4 and 5 are smaller and could ship earlier if scoped separately.

**Other deferred security hardening items (not yet scheduled):**
- Script execution timeout (per-policy, cooperative via PowerShell.Stop + timer, NOT Task.Run - context switch loses HttpContext / ThreadStatic Runspace.DefaultRunspace / ambient DatabaseSwitcher).
- Request-size cap (maxRequestLength / maxAllowedContentLength) scoped to the remoting handler location only - file-upload cmdlets must still work.
- Session / runspace count cap per identity (needs runspace-vs-session accounting design - a single session can spawn many runspaces via background jobs).
- Unbounded remoting parameter binding size cap (RemoteScriptCall.ashx.cs:1228-1264).
- Slowloris: recommended applicationHost.config snippet in deployment docs (SPE can't fix at code layer).
- _throttleState LRU eviction if non-global throttle scopes ever get enabled.
